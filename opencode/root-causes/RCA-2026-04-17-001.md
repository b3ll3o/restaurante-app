## 4. Causa Imediata

`createClient()` era chamado no nível do componente `AdminLayout`, gerando **uma nova instância a cada renderização**. Isso fazia com que `supabase.auth` fosse um objeto diferente a cada render, causando re-execução do `useEffect`.

---

## 5. Causa Raiz (5 Porquês)

| # | Pergunta | Resposta |
|---|----------|----------|
| 1 | Por que o useEffect re-executava infinitamente? | Porque `supabase.auth` era uma dependência que mudava a cada render |
| 2 | Por que `supabase.auth` mudava a cada render? | Porque `createClient()` era chamado diretamente no corpo do componente |
| 3 | Por que `createClient()` não estava memoizado? | O código não usava `useMemo` ou `useRef` para estabilizar a referência |
| 4 | Por que o código não seguia o padrão de estabilização? | Provavelmente esquecido durante implementação do PRD 008 |
| 5 | Por que não foi detectado antes? | Falta de teste E2E para o fluxo de login |

### Causa Raiz Final

O `createClient()` do Supabase era instanciado diretamente no corpo do componente sem `useMemo`, gerando uma nova referência de `supabase.auth` a cada render. Isso ativava o `useEffect` que verificava autenticação, disparando redirect para `/admin/login` repetidamente.

// app/admin/layout.tsx (e outros 8 arquivos)
import { useEffect, useState } from "react";

export default function AdminLayout({ children }) {
  const [isLoading, setIsLoading] = useState(true);
  const router = useRouter();

  // ❌ BUG: nova instância a cada render
  const supabase = createClient();

  useEffect(() => {
    const getUser = async () => {
      const { data: { user } } = await supabase.auth.getUser();
      if (!user) {
        router.push("/admin/login");
        return;
      }
      setIsLoading(false);
    };
    getUser();
  }, [router, supabase.auth]); // ⚠️ supabase.auth muda a cada render!
